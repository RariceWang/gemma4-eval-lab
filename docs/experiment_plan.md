# 实验方案

## 1. 研究问题

**Gemma 4 系列本地可运行版本在本地环境中的多语言推理能力与开放式生成质量是否存在稳定差异？如果存在，这些差异主要体现在哪些任务和语言上？**

## 2. 研究假设

- `H1`：`gemma4:26b` 在客观推理和开放式生成上整体优于 `gemma4:e4b`
- `H2`：模型规模越小，在低资源语言上的性能下降越明显
- `H3`：较大模型在开放式生成中的优势会比客观题更明显，但代价是更高延迟

## 3. 模型选择

选择理由：

- 同一模型家族，控制变量更容易
- 本地已可运行，不依赖额外 API
- 版本差异明确，适合做“系列评测”

默认本地模型列表：

| 模型 | 参数规模 | 上下文长度 | 预期定位 |
|---|---:|---:|---|
| `gemma4:e4b` | 8.0B | 131072 | 低成本、速度优先 |
| `gemma4:26b` | 25.8B | 262144 | 中档平衡 |

可选扩展模型：

| 模型 | 参数规模 | 上下文长度 | 说明 |
|---|---:|---:|---|
| `gemma4:31b` | 31.3B | 262144 | 能力优先，但 24G 内存机器不建议默认启用 |

## 4. 实验任务

### 4.1 多语言数学推理

- 数据来源：MGSM 子集或同风格自建题
- 语言：英文、中文、斯瓦希里语
- 指标：`Exact Match`
- 输出约束：只返回最终答案

### 4.2 多语言常识推理

- 数据来源：XCOPA 子集或同风格自建题
- 语言：英文、中文、斯瓦希里语
- 指标：`Accuracy`
- 输出约束：只返回 `A` 或 `B`

### 4.3 开放式生成质量

- 任务：中文摘要、英文学术改写、中文提纲生成
- 指标：人工评分
- 评分维度：
  - 内容完整性
  - 语言流畅性
  - 风格控制
  - 结构清晰度

建议采用 1 到 5 分量表，并至少两轮交叉复核。

## 5. 参数设置

为了保证公平性，默认统一使用：

- `temperature = 0.2`
- `top_p = 0.9`
- `num_ctx = 8192`
- 单轮推理，不做多次采样投票

控制原则：

- 所有模型使用相同提示模板
- 所有任务按固定顺序批量运行
- 客观题禁止输出解释，降低打分歧义

## 6. 评估指标

### 客观指标

- `Exact Match`
- `Accuracy`
- 平均响应时延
- 失败率

### 主观指标

- 摘要完整性
- 改写质量
- 语言自然度
- 结构清晰度

## 7. 数据组织方式

至少形成 3 组对比数据：

1. **模型维度对比**：`e4b` 与 `26b` 横向比较
2. **语言维度对比**：英 / 中 / 斯瓦希里语横向比较
3. **任务维度对比**：数学推理 / 常识推理 / 开放式生成横向比较

如果时间允许，可增加第 4 组：

4. **效率维度对比**：延迟与质量的权衡

## 8. 实验执行流程

1. 运行环境检查并记录
2. 用 `datasets/pilot/tasks.jsonl` 做冒烟测试
3. 扩展到正式数据集
4. 统计客观指标
5. 对开放式生成结果进行人工评分
6. 汇总为表格与图表
7. 按 3 页报告结构写作

## 9. 风险与应对

- 风险：本地长任务耗时过长或内存不足
  - 应对：先跑子集，再扩展全量
- 风险：`31b` 超出本机舒适运行范围
  - 应对：默认不启用 `31b`，仅作为扩展实验
- 风险：开放式生成难以自动量化
  - 应对：明确人工评分量表
- 风险：GitHub 远端创建能力受限
  - 应对：先完成本地仓库，确认远端创建方式后再推送
